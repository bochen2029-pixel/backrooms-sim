A read-only quality-control audit of the Backrooms Sim codebase against its stated
architecture / contracts / invariants, run as a multi-agent workflow with adversarial
verification of every high/critical finding.

  determinism/INV-1..8, spec-doc consistency, gate/test/golden coverage, Phase IV coherence),

  adversarial skeptic re-read the actual code to confirm or refute each high/critical finding.

  confirmed criticals.** The single "critical" that survived to verification (a claimed inverted

- **[CRITICAL | bug]** Shaft hole-cutting logic inverted: ceiling closed at entry, floor open at landing -- `gen/src/chunk.cpp:105-106`
  The shaftFloorOpen and shaftCeilOpen conditions are inverted. For a shaft spanning [top_level-depth, top_level], the current code closes the ceiling at top_level (where the void should OPEN so the wanderer can fall in) and opens the floor at (top_level-depth) (where they should LAND on a solid surface). The logic should be: shaftFloorOpen should be TRUE for levels EXCEPT the landing level (top_level - depth), and shaftCeilOpen should be TRUE for levels EXCEPT the entry level (top_level). Currently, both conditions use >= or <= in the wrong direction, making shaft voids non-functional and potentially trappable.
  *Fix:* Fix lines 105-106 in gen/src/chunk.cpp: change shaftFloorOpen condition to 'key.level > shaft.top_level - shaft.depth && key.level < shaft.top_level' (open all except landing), and shaftCeilOpen to 'key.level > shaft.top_level - shaft.depth && key.level <= shaft.top_level' (open all except entry). Add test coverage for shaft geometry (push/pop cells through the void at each level).  
  _[verified: REFUTED -> false-positive]_
- **[MEDIUM | inconsistency]** ChunkContentHash omits collision data, potentially missing changes -- `gen/src/chunk.cpp:212-229`
  ChunkContentHash hashes only render data (vertices: pos, nrm, color, uv, material) but not collision boxes. If a bug or feature change affected only collision geometry (e.g., wall thickness, pillar size) without altering vertices, the hash would remain identical, masking determinism breaks in collision replay. While the current test suite checks hash stability and distinctness by key/seed, it doesn't validate that collision changes affect the hash.
  *Fix:* Extend ChunkContentHash to include collision boxes, or document why collision is excluded (e.g., if it's derived deterministically from vertices). If collision is independent, add a separate collision_hash field or integrate all collision data into the main hash. Add a test that verifies collision-only changes update the hash (if collision should be covered).
- **[LOW | dead-code]** Dead function build_stairwell: declared but never called -- `gen/src/layout.cpp:154-169, gen/include/gen/layout.h:52-54`
  build_stairwell is implemented and declared in the public header but is not invoked anywhere in the module or tests. MODULE.md mentions it as a planned M7 collision-only set piece, but it's not integrated into GenerateChunk. This leaves dead code that could bitrot or confuse maintainers.
  *Fix:* Either (1) integrate build_stairwell into GenerateChunk as a planned feature with a clear ADR if still planned for M27+, (2) remove it if deemed out-of-scope, or (3) add a NOTE comment in the header explaining it's a planned future feature and mark it with [[maybe_unused]]. Clarify in MODULE.md or an ADR when/if collision-only descending stairs will be added.
- **[INFO | doc-drift]** Comment describes shaft void geometry semantics but code doesn't match -- `gen/src/chunk.cpp:101-103`

### stream module (stream/)

_The stream module implements a bounded chunk ring and background worker pool for procedural level streaming. Core components: StreamManager (chunk ring with 1-arg/2-arg/3-arg update modes per M28/M30), worker threads with two-queue synchronization (requests/ready), and main-thread residency management. The module correctly maintains INV-4 (bounded memory via ring size), supports M28 two-level and M30 multi-level residency without affecting the sim core (INV-1), and the 1-arg path is documented as bit-identical to pre-M28 behavior. However, critical race conditions exist in the `in_flight_` set management between the main thread and worker threads, violating thread-safety invariants and potentially causing chunks to be double-enqueued or incorrectly tracked._

- **[CRITICAL | contract-violation]** Race condition in enqueue(): in_flight_ accessed without synchronization -- `stream/src/stream_manager.cpp:36-43`
  The enqueue() method holds req_mtx_ only for lines 38-40 (inserting into requests_), then releases the lock before inserting into in_flight_ at line 41. Between the unlock and the insert, a worker thread can pop the request from the queue, call GenerateChunk(), and insert the result into ready_. The main thread then inserts into in_flight_, making the set inconsistent with the actual state (the chunk is already generated but is still marked as in-flight). This violates the invariant that in_flight_ accurately tracks pending generation work.
  *Fix:* Hold req_mtx_ for both the requests_.push_back() and in_flight_.insert() operations, or protect in_flight_ with a separate mutex (ready_mtx_ is for the ready queue, not the pending set). The safest fix is to extend the lock: {lock_guard lk(req_mtx_); requests_.push_back(k); in_flight_.insert(k);} then notify outside the lock.  
  _[verified: CONFIRMED -> high]_
- **[HIGH | bug]** TOCTOU race in update(): in_flight_ checked but enqueue() has unsynchronized insert -- `stream/src/stream_manager.cpp:89-91, 114-116`
  Both the 2-arg update(center, extra_level) at lines 89-90 and the 3-arg update(center, lo_level, hi_level) at lines 114 perform a check-then-act pattern: they read in_flight_ without holding any lock to avoid double-enqueueing, but then call enqueue() which itself modifies in_flight_ unsafely (per finding #1). If a worker thread completes a chunk between the check at line 90 and the call to enqueue() at line 91, the main thread will redundantly enqueue it again because in_flight_.erase() (line 71) happened on the worker thread, but the main thread's check already passed. Additionally, the reading of in_flight_ itself is a data race: no lock protects it.
  *Fix:* Either: (a) protect all in_flight_ accesses with a dedicated mutex, or (b) refactor enqueue() to atomically check and insert (returning bool for already-enqueued), or (c) tolerate duplicate enqueues and deduplicate at generation time. Option (a) is most robust: add std::mutex in_flight_mtx_; protect all in_flight_ reads and writes with lock_guard<mutex> lk(in_flight_mtx_).  
  _[verified: CONFIRMED -> high]_
- **[HIGH | bug]** Missing thread-safety documentation on public interface -- `stream/include/stream/stream_manager.h:27-28`
  The header declares update(), wait_idle(), resident(), and resident_count() as public, but provides no documentation of their threading contract. From code inspection, update(), resident(), resident_count(), and collect() (implicit, called by update()) are main-thread-only; wait_idle() is safe to call from any thread. This implicit contract is violation-prone — the user (e.g., app/src/main.cpp) is not warned that calling update() and resident() concurrently with a worker thread is unsafe, nor is the main-thread-only rule documented.
  *Fix:* Add thread-safety contract to the class and each public method. Example: 'ThreadSafety: ctor/dtor must be called on the main thread. update(), resident(), resident_count(), and collect() are main-thread-only. wait_idle() is thread-safe. Workers are spawned in ctor and joined in dtor; no manual thread management is exposed.'  
  _[verified: CONFIRMED -> high]_
- **[MEDIUM | inconsistency]** collect() is private but used implicitly in update(); lack of atomic region clarification -- `stream/src/stream_manager.cpp:64-75`
  The collect() method is implicitly called at the end of update() (line 96, 120) to gather completed chunks. However, there is no comment or documentation explaining that the enqueue-then-collect sequence is the intended synchronization pattern. The code does not document that between the check at lines 89-90 and the call to enqueue(), a chunk could theoretically complete and be collected on the worker thread, leaving in_flight_ in an inconsistent state.
  *Fix:* Add a comment block above update() explaining the synchronization: 'Main-thread only. Enqueues missing ring chunks, then collects any that finished, then evicts out-of-ring. The check-then-enqueue loop is safe only because no collect() happens between the check and enqueue for the SAME chunk.' Also document the invariant: 'A chunk is in in_flight_ iff it is on the requests_ queue or being generated.'
- **[MEDIUM | inconsistency]** 3-arg update() swaps parameters silently; contract unclear on invocation order -- `stream/src/stream_manager.cpp:106-107`
  The 3-arg update(center, lo_level, hi_level) swaps lo_level and hi_level if lo_level > hi_level (line 107), but the caller (app/src/main.cpp:1027) does not appear to rely on or document this auto-correction. The swap is correct for the eviction logic (line 125), but the silent parameter re-ordering is an implicit contract that could mislead maintainers or lead to subtle bugs if the calling code expects the levels in a specific order.
  *Fix:* Either: (a) add a precondition assertion (assert(lo_level <= hi_level)) to enforce caller discipline, or (b) add a comment in the method explaining the swap and why the caller should have ordered them correctly. Also document in the header: 'param lo_level, hi_level: Level range [lo_level, hi_level] inclusive. If lo_level > hi_level, they are swapped.'
- **[MEDIUM | bug]** Data race on in_flight_ without mutex or documentation -- `stream/src/stream_manager.cpp:41`
  Line 41 writes to in_flight_ (a std::set) without holding any lock. The std::set is not thread-safe; concurrent access from the main thread (in update()) and implicit access from worker threads (via enqueue()) is a data race. The line is unprotected and unsynchronized, and no comment explains why it is safe (it is not).
  *Fix:* Wrap the in_flight_.insert(k) with a dedicated mutex protecting all in_flight_ accesses, or merge the operation into the req_mtx_ lock.
- **[LOW | doc-drift]** MODULE.md does not mention M28/M30 updates in its description -- `stream/MODULE.md:1-3`
  The MODULE.md lists the 1-arg update() in the 'Public surface' section (line 9) but does not mention the 2-arg or 3-arg overloads added in M28/M30. The doc is incomplete.
  *Fix:* Update MODULE.md to list all three update() signatures: 'update(ChunkKey center) — single-level ring (delegates to 2-arg with center.level); update(ChunkKey center, int32_t extra_level) — two-level ring (M28); update(ChunkKey center, int32_t lo_level, int32_t hi_level) — multi-level range (M30).'
- **[LOW | inconsistency]** 1-arg update() delegates to 2-arg but does not document the bit-identity guarantee -- `stream/src/stream_manager.cpp:77`
  Line 77 implements the 1-arg update(center) by delegating to update(center, center.level), which correctly implements the M28 bit-identity guarantee ('extra_level == center.level is exactly the prior single-ring behaviour' per comment at line 81-82). However, the 1-arg method itself has no comment explaining this contract or referencing M28/ADR-054.
  *Fix:* Add a comment above line 77: '// Single-level ring: delegates to the 2-arg path with extra_level==center.level. This // implements the M28 bit-identity guarantee; old code paths using update(center) alone // produce identical results (ADR-054).'

### render_dxr module audit

_The render_dxr module implements DXR 1.1 path-traced mode for the Backrooms simulation. It builds per-chunk BLASes and a TLAS from streamed geometry, supports M25 dynamic creature animation via cached chunk BLASes and per-frame creature BLAS updates, and provides both primary-ray and fully path-traced rendering with temporal accumulation. The module is self-contained (own Device5), uses inline RayQuery for PT shading with emissive fluorescent lighting, and integrates depth comparison gates. The codebase demonstrates correct synchronization with wait_idle() calls before GPU fence operations and proper UPLOAD heap mapping patterns. Acceleration structure lifetimes are preserved through ComPtr RAII; the shade buffer uses InstanceID-based vertex indexing for PT shader lookups across concatenated chunk geometry. The implementation is largely sound for single-level operation but has overlooked edge-case issues around 24-bit InstanceID overflow limits and multi-level chunk handling (M28+)._

- **[HIGH | bug]** InstanceID 24-bit overflow unchecked: silent data corruption on >16M vertices -- `render_dxr/src/dxr.cpp:780, 1008`
  The InstanceID field in D3D12_RAYTRACING_INSTANCE_DESC is 24 bits (0xFFFFFF = 16777215). Line 780 masks vtxOffset with `& 0xFFFFFFu` to store the shader's vertex-buffer start index, and line 1008 does the same for the creature. However, there is NO validation that the total concatenated vertex count (allVerts.size() + kMaxCreatureVerts) stays within 24 bits. If a ring of resident chunks (e.g., 169 chunks at ~100k verts each) exceeds 16M vertices, vtxOffset wraps silently, causing two TLAS instances to share the same InstanceID—the shader will read geometry from the wrong buffer region, producing silently corrupted depth/shading. The mask operation is correct in principle (DXR hardware enforces 24-bit), but the lack of error checking against the limit violates the contract that DXR and raster renderers should be bit-comparable (INV-7).
  *Fix:* Add a pre-build check: `if (allVerts.size() > 0xFFFFFFu) { last_error_ = "too many vertices for InstanceID (24-bit limit)"; return false; }` after line 790 (after building allVerts but before creating the TLAS). Document the limit in a comment near kMaxCreatureVerts. This is a structural limit of DXR, not a tuning parameter, so catching it early prevents hard-to-debug corruption on large worlds.  
  _[verified: CONFIRMED -> high]_
- **[MEDIUM | inconsistency]** M28 multi-level rendering not documented; no per-level BLAS caching strategy -- `render_dxr/MODULE.md:1-5, render_dxr/src/dxr.cpp:720-944`
  The stream module's update(ChunkKey, extra_level) and update(ChunkKey, lo_level, hi_level) methods (stream/include/stream/stream_manager.h:37-48) enable M28+ multi-level residency. However, render_dxr/MODULE.md mentions only M9 phase 2b (single-level BLAS/TLAS) and makes no reference to M25-M28 milestones. The build_scene function (line 720) concatenates all resident chunks into a flat vertex buffer and builds one global TLAS; it does not detect or segregate chunks by level. This is functionally correct (chunks at different Y levels are still collided correctly in world space), but the lack of documented strategy for handling M28's multi-level rings is an inconsistency: if future optimizations (e.g., per-level BLAS caching across frame boundaries) become desired, the contract boundary is unclear. The module should explicitly state: 'build_scene accepts ResidentChunk(s) from any mix of levels and builds a flat TLAS without level-specific optimizations.'
  *Fix:* Update render_dxr/MODULE.md to note: (1) Current (M9-M25) behavior: flat concatenation of all resident chunks, regardless of level; (2) M28 readiness: build_scene is compatible with multi-level rings but does not cache per-level BLASes; (3) future multi-level BLAS caching would require stream/render interface changes (e.g., grouping chunks by level in ResidentChunk list).
- **[MEDIUM | contract-violation]** shadeVb coherence assumptions undocumented: UPLOAD heap CPU/GPU synchronization not explicit -- `render_dxr/src/dxr.cpp:796-803, 964-967`
  shadeVb is allocated as D3D12_HEAP_TYPE_UPLOAD (line 797) and mapped once in build_scene (line 801), updated in place, and unmapped (line 803). Later, in update_creature (line 963-966), shadeVb is remapped, the creature tail is modified via memcpy, and unmapped again. D3D12_HEAP_TYPE_UPLOAD heaps guarantee host coherence, but the code assumes that CPU writes via memcpy followed by Unmap are immediately visible to GPU reads without explicit memory barriers. The wait_idle() at line 955 ensures GPU has finished prior work, but there is no explicit cache flush or UAV barrier between Unmap and the GPU's next render_pt invocation. On systems with weak GPU memory models or aggressive L1 caching, this could theoretically cause stale reads. The code is likely correct (UPLOAD is host-visible), but the assumption is undocumented, making it fragile across GPU architectures.
  *Fix:* Add a comment at line 801 and line 963 documenting the coherence assumption: 'D3D12_HEAP_TYPE_UPLOAD guarantees host coherence; CPU writes via memcpy are visible to GPU reads immediately after Unmap (no explicit D3D12_RESOURCE_BARRIER_UAV needed).' Optionally, for absolute safety on weak-coherence hardware, add a D3D12_RESOURCE_BARRIER(UAV, shadeVb) after each Unmap, though this is likely unnecessary and would add latency.
- **[LOW | dead-code]** creatureVb allocated fresh every frame in update_creature: per-frame allocation waste -- `render_dxr/src/dxr.cpp:971-976`
  Line 971 calls make_buffer(...) to allocate a new creatureVb of size (count * sizeof(ChunkVertex)), then immediately maps and copies the creature geometry. This happens every frame M25+ calls update_creature. The allocation is unnecessary: a single creature VB could be pre-allocated at build_scene time (sized to kMaxCreatureVerts) and reused, or even allocated once in init() and resized on demand. The current design (allocate-per-frame) works but causes unnecessary GPU memory churn and allocation overhead. The D3D12_HEAP_TYPE_UPLOAD heap can handle the reuse pattern, and it would reduce per-frame latency.
  *Fix:* Refactor: allocate creatureVb once in build_scene (sized to kMaxCreatureVerts), store it in Impl, and reuse it in update_creature. Alternatively, pre-allocate in init() and skip allocation if size is sufficient. This is a performance micro-optimization, not a correctness issue, so prioritize lower than the overflow check.
- **[LOW | doc-drift]** InstanceID indexing comment mismatch: does not clarify flat triangle-list assumption -- `render_dxr/src/dxr.cpp:624, 730`
  Line 730 states: 'each instance's InstanceID is its start offset into that buffer, so (InstanceID + 3*PrimitiveIndex) indexes a vertex.' This is correct: the HLSL line 624 does `uint base = q.CommittedInstanceID() + 3u * q.CommittedPrimitiveIndex();` to index into g_verts (a StructuredBuffer of Vertex). However, the comment does not clarify that this formula assumes allVerts is a flat triangle list (3 verts per tri), which is the case (line 787 concatenates chunk triangles in order). The comment should note: 'Each chunk's vertices are concatenated in order; within a chunk, triangles are indexed [0..3), [3..6), etc.; InstanceID points to the chunk's first vertex, so (InstanceID + 3*PrimitiveIndex) finds the primitive's first vertex.'
  *Fix:* Clarify the comment at line 730 to: 'Each instance's InstanceID is its start vertex offset in the concatenated buffer. Triangles are stored as consecutive 3-vertex groups; the shader computes (InstanceID + 3*PrimitiveIndex) to find a triangle's first vertex. Within a chunk's BLAS, PrimitiveIndex is the local index into that chunk's geometry.' This removes ambiguity for future readers and M28+ multi-level scenarios.

### audio module (procedural synth + room-probe reverb + footsteps + M24 formant TTS)

_The audio module (L2) implements procedural synthesis with 60 Hz mains hum + harmonics, HVAC drone, footsteps, and Freeverb-style reverb. M6 provided deterministic offline WAV rendering via seeded PCG64 RNG and stateful Synth with phase-tracked oscillators. M14 added real-time playback via miniaudio + a lock-free SPSC float ring (producer=mixer, consumer=device callback). Determinism (INV-1) is preserved: each Synth instance is seeded independently, phases are double-precision and wrapped at 2π, and all state is local. The ring uses correct acquire/release synchronization for SPSC. Room probe pure rays against AABB walls to derive reverb RT60. Offline WAV is bit-identical x2 across runs; real-time path is presentation-only (not gated on determinism). Overall health is strong: synthesis is proven deterministic by unit test, WAV I/O round-trips exactly, room-probe sizing logic is sound, and the real-time mixer thread never blocks the sim tick. One minor design inconsistency noted: M14's AudioListener creation leaves pos/yaw zero (currently unused but violates contract)._

- **[MEDIUM | inconsistency]** Memory ordering asymmetry in readable_frames() -- `audio/include/audio/ring.h:32-36`
  The readable_frames() method loads w_ with acquire and r_ with relaxed. If this const method is intended as a public query usable from any thread, both loads should use acquire to ensure both cursors are synchronized. However, in practice it is only called from the producer thread (in writable_frames() path), where the producer owns w_ (relaxed is sufficient for self-read) and needs to see consumer's r_ (acquire needed). The current code is conservative but asymmetric. If any observer thread or other code path calls readable_frames(), a race could theoretically occur (unlikely in practice but violates the lock-free contract's symmetry assumption).
  *Fix:* Either (a) use acquire for both loads in readable_frames() for defensive correctness, or (b) document that readable_frames() and writable_frames() are producer-thread-only and update callers if any exist outside engine.cpp. Option (a) is safer for a public API and adds minimal overhead (just one extra acquire fence).
- **[LOW | drift]** Phase wrapping at kTwoPi vs. true 2π -- `audio/src/synth.cpp:84-90`
  The hum oscillators maintain double-precision phases and wrap at kTwoPi = 6.283185307179586 (an approximation to 2π). Over very long sessions, the phase might drift slightly relative to ideal 2π due to floating-point representation differences. For a 12-hour soak, this would cause negligible frequency error (<1 ppm), imperceptible in audio, but theoretically the hum is not perfectly at 60.0000 Hz over infinite time. The wrapped-phase design is standard and acceptable, but documentation could note this bound.
  *Fix:* No action required for v1.0; the drift is inaudible and acceptable per ARCHITECTURE §9 NFRs. For reference: sin(phase) has period 2π regardless of the representation; wrapping at the same constant (kTwoPi) ensures determinism; and double-precision carries ~15 significant digits, giving <1 ppm drift over 12 hours of audio. If sub-ppm accuracy is needed in future (post-v1.0), use fmod() or a modular arithmetic approach.
- **[LOW | inconsistency]** PCM16 conversion asymmetry (positive/negative range) -- `audio/include/audio/wav.h:82-91`
  The to_pcm16() function clamps float to [-1,1], then multiplies by 32767.0f and casts to int. This gives asymmetric int16_t range: +1.0f → 32767, but -1.0f → -32767 (not -32768, which is the int16_t minimum). The discrepancy is 1 LSB and inaudible (<0.003% amplitude error), but the range is technically not symmetric. ### Phase IV Headless Proof Modes (run_descend, run_ascend, run_shaftfall, run_abyss, run_vstream)

_The five Phase IV proof modes demonstrate vertical mechanics (ascent/descent, shafts, vertical streaming) through deterministic scripted scenarios. All modes correctly use seeded RNG, collision, and streaming managers, with determinism verified via bit-identical hash runs. Camera poses are correctly constructed, and level mathematics (level_base_y, level_from_y) are consistent. The modes largely fulfill their stated purposes: descend proves fall-to-sublevel, ascend proves stairwell climbability, shaftfall proves soft-catch landing, abyss and vstream prove multi-level visibility with bounded residency. - **[MEDIUM | contract-violation]** run_shaftfall: floating-point level boundary in landed assertion -- `app/src/main.cpp:2882`
  The soft-catch fall test asserts `endLevel == (sh.top_level - sh.depth)` where `endLevel = level_from_y(s.wanderer.pos.y)` and `level_from_y(y) = floor(y / 4.0)`. If swept collision resolves the wanderer to Y = -40.01 (just below a level-boundary floor at Y = -40), then floor(-40.01 / 4.0) = floor(-10.0025) = -11, failing the assertion even though the collision semantically landed correctly. The M26 contract requires determinism (INV-1) but not sub-pixel-perfect positioning; however, the gate detects this as failure rather than accepting a 'nearly landed' state. Risk: spurious failure if collision implementation places the wanderer slightly below rather than on a floor.
  *Fix:* Add a small epsilon to the level check: `endLevel >= sh.top_level - sh.depth - 1` to accept landings within one level of the target, or use a Y-space tolerance: `s.wanderer.pos.y >= botY - 0.1f` instead of level_from_y.
- **[MEDIUM | contract-violation]** run_vstream: hard-coded residency ratio assumes full ring coverage -- `app/src/main.cpp:2842`
  The test asserts `res2 == res1 * 2`, expecting exactly 2x resident chunks when a second level is added. This assumes the streaming ring fills completely (all (2*radius+1)^2 chunks present) at both levels. However, if the shaft/stair is positioned near the origin or another boundary, one level's ring could be partially clipped. Additionally, the render loop (lines 2813-2815) has a fixed iteration limit of 1200 renders; if not all chunks are drawn before the limit, the diff is artificially small and could pass the `see_through > 0.5` threshold spuriously. The gate does NOT verify that resident_count actually equals the expected value — it only checks the computed diff.
  *Fix:* Either (1) relax the assertion to `res2 >= res1 * 2 - epsilon` with a documented tolerance, (2) search for stairs far from boundaries (e.g., require cx/cz to be in [-30, 30] range), or (3) verify that the render loop completes: assert `drawn == target` after rendering.
- **[MEDIUM | contract-violation]** run_abyss: hard-coded residency formula without boundary margin -- `app/src/main.cpp:2974`
  The abyss test asserts `resDeep == resShallow * (renderDepth + 1)`, expecting a precise (renderDepth+1)x multiplier. This assumes the streaming ring fills completely for each of renderDepth+1 levels. If the chunk grid has a boundary effect or if renderDepth is clamped (line 2928: `renderDepth = min(sh.depth, 4)`), the residency could differ. - **[LOW | inconsistency]** run_ascend: climb threshold hard-coded without justification -- `app/src/main.cpp:2755`

  *Fix:* Raise the threshold to `climb >= 3.1f` (well above ceiling) to ensure entry into the next level, or use `level_at_max >= 1` instead, matching the `level_at_max` telemetry already printed on line 2763.
- **[LOW | smell]** vstream and abyss render loops use magic iteration limits without explanation -- `app/src/main.cpp:2813-2815, 2941-2943`

### PHASE IV Vertical System (M26-M30) Audit

_The PHASE IV vertical system is largely well-designed and coherent, with the level-implicit architecture (deriving level from Y position via `level_from_y`) preserved deterministically and INV-1 intact. M26 establishes multi-level streaming with a per-level collision gather (3×3 chunks at wanderer's floor). M27 implements procedural stairs via a hybrid density+backstop approach with aligned floor/ceiling holes using a shared seam hash (mirroring `door_index`). M28 adds two-level residency streaming for see-through at stairwells. M30 implements rare, deep open shafts using the same per-column architecture, with soft-catch falls via existing swept-AABB collision. The streaming manager correctly bounds residency at 2x and Nx the ring. The M29 Shoggoth confinement is enforced by structure (no vertical motion code, level derived from fixed Y at each step) rather than explicit guards. The per-level fluorescent light placement correctly adds `level_base_y(level_from_y(camera.y))` to the hardcoded `kCeilingHeight` return, yielding correct ceiling light positions. The acute risk is unguarded **stair+shaft cell collision**: when both a procedural stair and a rare shaft occupy the same cell at the same (cx, cz), both geometries are rendered (stairwell inside void), corrupting the cell. This is extremely rare (~1/19,500 probability per cell) but deterministic and will eventually occur in an infinite world. A secondary observation: the lighting discontinuity at level seams is avoided by stairwell geometry aligning seamlessly (riser tops at next level's floor), so lights transition cleanly._

  When a procedural up-stairwell (M27) and an open shaft (M30) occupy the same cell at the same chunk column (cx, cz), both geometries render: the stairwell riser boxes and floor/ceiling are generated inside what should be a void.
