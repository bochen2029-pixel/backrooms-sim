## ADR-001 .. ADR-009
See ARCHITECTURE.md §8 for the accepted set; expand each here as the build touches it. New dependencies, golden updates, and gate-threshold changes REQUIRE a new ADR entry in the same commit (Iron Rules 6 and 8).

## ADR-010 — vcpkg acquisition: pinned baseline + self-bootstrapping location
- **Status:** Accepted (M0).
- **Context:** ADR-007 chose vcpkg manifest mode. A fresh clone must resolve deps (Catch2, stb) with no manual setup, reproducibly.
- **Decision:** `vcpkg.json` pins `builtin-baseline` to `d592849579fb1fb22f87406b2184522ea21a8783`. `scripts/lib/common.ps1::Ensure-Vcpkg` discovers vcpkg in order `$env:VCPKG_ROOT` → `C:\vcpkg` → repo `extern/vcpkg`, cloning+bootstrapping (shallow) into `extern/vcpkg` (gitignored) when none exists. The dev machine uses `C:\vcpkg`.
- **Consequences:** Fresh clone self-bootstraps; vcpkg is never vendored in git; the baseline pin makes dependency versions reproducible (bumping it is a deliberate edit + ADR).

## ADR-011 — x64-windows-static triplet and static CRT
- **Status:** Accepted (M0).
- **Context:** The gate scripts run `app`, `tools`, and `tests` executables directly from `build/bin` and must not depend on DLL deployment.
- **Decision:** `VCPKG_TARGET_TRIPLET = x64-windows-static`; `CMAKE_MSVC_RUNTIME_LIBRARY = MultiThreaded$<$<CONFIG:Debug>:Debug>` (`/MT`). Third-party (Catch2, stb) and our code all link the static CRT.
- **Consequences:** Self-contained executables, no DLL-copy step, simpler gates; slightly larger binaries; CRT-mismatch is structurally impossible.

## ADR-012 — "test-the-gate" canary mechanism
- **Status:** Accepted (M0).
- **Context:** The M0 exit gate must prove a deliberately failing test is detected (would block a commit) without making the normal `ctest` run red.
- **Decision:** `gate_canary` is a Catch2 executable containing a `FAIL`, always compiled, registered in CTest as `DISABLED`. `scripts/gate.ps1` runs the binary directly and asserts a nonzero exit; the git `pre-commit` hook runs `quickcheck` (build + tests) so any real failing test blocks commits.
- **Consequences:** The normal suite stays green (canary disabled); the gate still mechanically proves failure detection; no reconfigure/second build dir needed.

## ADR-013 — warnings-as-errors scoping via external-header flags
- **Status:** Accepted (M0).
- **Context:** `/W4 /WX` on third-party headers (Catch2, stb) would fail the build on warnings we do not own.
- **Decision:** Strict flags live on the `backrooms_flags` INTERFACE target (our code only). Angle-bracket/`SYSTEM` includes are demoted with `/external:anglebrackets /external:W0`; the stb implementation TU is isolated in `br_stb` and compiled `/w`. The deterministic sim core additionally gets `/fp:strict` via `backrooms_sim_flags`.
- **Consequences:** `/WX` protects our code without third-party noise; "zero warnings-as-errors violations" is enforced by a clean build, not a grep.

## ADR-014 — M1 memory-soak metric: process private bytes, not the CRT debug heap
- **Status:** Accepted (M1).
- **Context:** The M1 gate wording is "60 s soak: stable memory (CRT debug heap delta ≈ 0)". We ship the **release** static CRT (`/MT`, ADR-011), which has no CRT debug heap, so that exact metric is unavailable without a separate debug-CRT build.
- **Decision:** Measure process **PrivateUsage** via `GetProcessMemoryInfo` (`PROCESS_MEMORY_COUNTERS_EX`) at soak start vs end and assert the delta `< 16 MiB` over the 60 s headless soak. "No fence timeouts" is enforced structurally: the renderer returns failure (nonzero process exit) if any fence wait times out.
- **Consequences:** Build-config-independent leak detection that preserves the gate's intent (flat memory, zero slope). Empirically ~1.5 MB over 62k frames in 10 s — warmup-dominated, no per-frame growth. A real leak in the tight render loop would be hundreds of MB and trip the gate immediately.

## ADR-016 — M2 collision model: AABB-proxy capsule, per-axis swept + substepped
- **Status:** Accepted (M2).
- **Context:** The wanderer is canonically a "capsule" (glossary). The world is entirely axis-aligned (Backrooms). The exit gates require no wall penetration at any speed, sliding that preserves tangential velocity, and no floor tunneling under large per-tick movement.
- **Decision:** Model the wanderer's collision shape as an **AABB proxy** with half-extents `(radius, halfHeight, radius)`. `move_and_collide` advances the box and resolves **one axis at a time** (X, then Z, then Y), zeroing only the blocked axis (sliding preserves the others). Movement is **substepped** (≤ 0.05 m/substep, capped at 256) so fast motion cannot tunnel; the proxy's 0.7 m width also exceeds any substep, so it can never skip a solid.
- **Consequences:** Deterministic, robust, and exactly satisfies the three collision gates for axis-aligned geometry. Corner behaviour is square rather than rounded; a true rounded-capsule closest-point test can refine this in a later milestone without changing the contract.

## ADR-017 — M2 boundary contracts: world_view_v1 + replay_v1
- **Status:** Accepted (M2).
- **Context:** ARCHITECTURE.md §5 lists these boundaries; M2 is the first milestone that crosses them (core → renderer snapshot; input stream serialization).
- **Decision:** `contracts/world_view_v1.h` (`CameraPose`, `BoxInstance`, `WorldView` with `std::span` of boxes) and `contracts/replay_v1.h` (`InputCommand`, `ReplayHeader`, magic/version) are added as a header-only `contracts` INTERFACE target. ## ADR-019 — M3 streaming architecture: pure world-coord chunks + ring + worker pool
- **Status:** Accepted (M3).
- **Context:** Need an unbounded, never-repeating, bounded-memory world that streams around the wanderer (INV-2/3/4).
- **Decision:** `GenerateChunk(seed, key)` (chunk_gen_v1) is pure/total: it seeds a `Pcg64` from `hash(seed, key)`, emits placeholder geometry (grid floor + interior posts) in **world coordinates** so adjacent chunks share boundary vertices (seam agreement without neighbor queries). `stream::StreamManager` keeps a square `(2r+1)^2` ring around a center: missing chunks are generated on a background worker pool, the main thread collects results and evicts chunks outside the ring (bounded residency, INV-4). Streaming is fully decoupled from the sim, so it cannot perturb determinism (INV-1). The streaming walk collides against `core::open_ground()` (an implicit floor), keeping `core` free of any `gen`/`stream` dependency.
- **Consequences:** Far-from-origin chunks (|coord| beyond ~16M m) lose float precision; camera-relative rendering is deferred until needed. Real Level-0 rooms/connectivity replace the placeholder in M4.

## ADR-020 — M3 chunk vertex-buffer pool + warmup (hitch-free uploads)
- **Status:** Accepted (M3).
- **Context:** Per-chunk `CreateCommittedResource` on stream-in caused ~3 ms upload spikes (p99 ≈ 3x median) and a 32 ms first-frame pipeline-compile hitch.
- **Decision:** The renderer keeps a pool of persistently-mapped upload buffers (`kChunkSlotCapacityBytes` each), reused across chunks via a free-list; steady-state stream-in is a `memcpy` only (no allocation). The app runs an untimed **warmup** that builds the pipeline and fills the initial ring before any measured frame. A per-frame upload budget bounds work.
- **Consequences:** After warmup the pool size is stable; p99/median ≈ 1.2x at 1280×720. GPU memory bounded by `(2r+1)^2` slots.

## ADR-021 — M3 hitch + soak gate metrics
- **Status:** Accepted (M3).
- **Context:** MILESTONES M3 says "no frame > 2x median"; NFR §9 (the authoritative source for gate numbers) says **p99 frame < 2x median**. A headless loop at ~760 fps has unavoidable OS-scheduling jitter in the top 1%.
- **Decision:** The hitch gate walks 100+ chunks at **2560×1440** (the NFR's target resolution, where per-frame work dwarfs fixed OS jitter) and asserts **p99 frame-time < 2× median** (NFR §9). Three robustness measures keep it non-flaky without weakening the threshold: (1) 1440p raises the median to ~2.7 ms so the jitter tail is a small fraction; (2) the app raises the OS timer resolution to 1 ms (`timeBeginPeriod`) so GPU fence-wait wakeups pace smoothly; (3) the gate takes the **best of 2 walks** — a real streaming hitch repeats, a one-off OS load spike does not. Empirically p99/median ≈ 1.3–1.6× (incl. immediately post-build). The memory gate measures process private-bytes delta over a soak (default **600 s**, parameterized via `-StreamSoakSeconds`) and asserts `< 16 MiB` growth (private-bytes proxy per ADR-014).
- **Consequences:** Robust, non-flaky gate aligned with the NFR. An earlier 1280×720 single-run variant flaked under post-build load (p99 2.35×); the three measures above fixed it. Regression sweeps can pass a shorter `-StreamSoakSeconds`; the milestone-green tag uses the full 600 s.

## ADR-022 — M4 Level-0 generator: spanning-tree maze + edge-hash doorways
- **Status:** Accepted (M4).
- **Context:** Each chunk must be internally traversable with no sealed regions (INV-3), generated independently (INV-2), yet agree with neighbours on shared-edge doorways without communication.
- **Decision:** A chunk is a G=8 cell grid (`gen/layout.h`). `generate_layout` builds a **spanning tree** over the cells (iterative recursive-backtracker) — provably connecting all cells — then carves ~25% extra openings for loops/rooms, then opens one doorway on **each** of the 4 edges at a cell index from `door_index(seed, sharedCoords, axis)`; a vertical edge between chunks `cx-1|cx` uses `cx` so both sides compute the same index. Each chunk emits its full perimeter (both sides of a shared edge place identical, aligned walls + doorway). `GenerateChunk` builds floor + wall geometry (render verts + collision `BoxInstance`s) in world coordinates. `validate_connectivity` (flood-fill, zero sealed) and `ValidateChunkGeometry` (no degenerate/floating/fat/stacked walls) gate 10,000 chunks each.
- **Consequences:** Connectivity is structural (never sealed). The geometry validator uses thresholds well between wall thickness (0.3 m) and cell size (4 m) so far-chunk float noise (deferred camera-relative rendering, ADR-019) doesn't false-fail. Real rooms/biomes layer on in M7.

## ADR-023 — M4 collision integration + walk-bot v1
- **Status:** Accepted (M4).
- **Context:** The wanderer must collide with generated walls, but `core` cannot depend on `gen`/`stream` (DAG). The walk-bot must traverse the maze and detect being sealed in.
- **Decision:** The **app** (composition root) gathers wall AABBs from the 3×3 chunks around the wanderer (regenerated via `GenerateChunk` only on chunk crossing — deterministic, no streaming dependency) plus an implicit ground floor, and passes them to the 3-arg `core::tick`. The walk-bot (`--walkbot`) is a seeded **wander + escape-on-block** controller (sweep the heading when displacement stalls). **Stuck detection** measures the position's bounding-box extent over each 10 s window (spec: "position variance ≈ 0" = motionless/sealed), NOT net displacement — a wanderer that thrashes (slides/turns without net progress) still ranges far and is not stuck.
- **Consequences:** `core` stays gen-free; the walk-bot is deterministic (reproducible WorldState hash) and completes 1 km with zero stuck events across 5 seeds. Wall-following / smarter navigation is a future refinement.

## ADR-024 — M4 top-down debug golden
- **Status:** Accepted (M4).
- **Context:** The M4 gate needs a per-seed top-down render of a 3×3 chunk block.
- **Decision:** `render_topdown` uses a hand-built **orthographic** MVP (world XZ→screen, world Y→depth) over `[c-half, c+half]²`, drawing the chunk pipeline with a straight-down light. `--topdown` generates chunks (0..2)² and renders at 512×512. Goldens `goldens/m4/topdown_seed{1,7}.png` captured via `goldgen`; the gate renders ×3 bit-identical and compares per seed.
- **Consequences:** Committed alongside this ADR (INV-8). Distinct per-chunk tints + maze walls make connectivity/seams visually auditable. Changing geometry, tints, or the projection requires a `goldgen` update + ADR.

## ADR-025 — M5 procedural materials + GPU texture-array path
- **Status:** Accepted (M5).
- **Context:** The backrooms look needs material textures (yellow wallpaper, damp carpet, ceiling tiles, fluorescent panels, baseboards) with no asset files (everything procedural) and without breaking chunk determinism (INV-2) or core isolation (INV-5).
- **Decision:** `render_d3d12/texgen.{h,cpp}` synthesises 5 materials (`TexKind` Wallpaper/Carpet/CeilingTile/Fluorescent/Baseboard) as 256² RGBA from a per-pixel hash of `(kind, seed, x, y)` — pure, **D3D12-free**, unit-tested (`test_texgen`), deterministic per `(kind, seed)` with a `texture_hash`. `ChunkVertex` gains `uv` + `material` (kMat* ids that index the texture-array slice; 48-byte stride). `GenerateChunk` assigns floor=Carpet, walls=Wallpaper, and emits a **ceiling** grid (render-only, not collidable) with a regular fluorescent-tile pattern from `is_fluorescent_cell` (shared verbatim by gen geometry and the renderer's lights). The renderer uploads the 5 slices once into a **Texture2DArray** via an upload heap + `CopyTextureRegion` (fenced), bound through a shader-visible **SRV heap** + **static sampler** + root descriptor table; the lit PSO samples by `(uv, material)` and applies a per-chunk hue tint.
- **Consequences:** No asset files; textures regenerate identically anywhere. The contract grows (`uv`/`material` in `chunk_gen_v1.h`, `BoxInstance` hoisted to `geometry_v1.h`), and `ChunkContentHash` now covers them, so M3 regen/seam goldens were re-verified. `render_topdown` discards ceiling/fluorescent via a root-constant toggle so the M4 top-down goldens stay bit-exact (no re-capture).

## ADR-026 — M5 forward fluorescent lighting + deterministic flicker
- **Status:** Accepted (M5).
- **Context:** Fluorescent lighting is the backrooms' defining mood, including the occasional buzzing/flickering tube. It must be **replayable**: a recorded (seed, input) stream must reproduce identical lighting, so randomness cannot live in the renderer or use wall-clock.
- **Decision:** Flicker lives in **`core`** — `light_flicker(world_seed, light_id, tick)` is pure, `/fp:strict`, branch-free value-noise: ~1/8 of lights flicker within `[0.35, 1.0]` over ~24 Hz steps, the rest are steady (1.0). `render_chunks(camera, resident, budget, tick, out)` gathers the ceiling-grid fluorescent lights within R=10 cells of the camera into a CBV (b1, ≤64 lights), each position from the shared `fluorescent_light_pos` and intensity scaled by `light_flicker`. The lit shader sums per-light Lambert × distance attenuation `i/(1+0.09d+0.05d²)` over a flat ambient (0.22), then applies a **highlight knee** (compress everything above unity exposure by 0.18×) so stacked lights keep the wallpaper's yellow hue instead of clipping to pure white.
- **Consequences:** Lit replays are bit-reproducible from (seed, tick) with no renderer-side RNG; `core` stays graphics-free (INV-5). The knee + ambient/intensity (0.22 / 1.0) were tuned against the reference look (even, hazy, slightly over-exposed, never blown-out). Clustered/path-traced lighting parity is a later milestone (M6+).

## ADR-027 — M5 lit-render gate: fixed-pose goldens + luminance band + FPS
- **Status:** Accepted (M5).
- **Context:** M5 changes pixels, so it needs golden coverage of the textured+lit output plus guards that the frame is neither all-black nor blown-out and that lighting didn't blow the frame-time budget.
- **Decision:** `app --shot --pose P [--seed S] [--ticks T] --out p.png` renders one of **5 canonical camera poses** (forward/right/back at the proven-open spawn cell, plus look-up at the ceiling fluorescents and look-down at the carpet) at a fixed flicker tick — bit-exact per `(seed, pose, tick, GPU)` despite multithreaded streaming (co-planar same-material geometry is draw-order-invariant). It also prints a **luminance histogram** (`luma_mean`, `luma_p01/p50/p99`, `frac_black`, `frac_white`). `gate.ps1 M5` checks: **5 poses × 3 seeds** (1, 7, 42) goldens `goldens/m5/shot_seed{S}_pose{P}.png` (640×360, captured via `goldgen capture`) — bit-identical across two fresh processes, golden-matched, debug-clean, and in the luminance band (mean ∈ [50, 220], frac_black ≤ 0.35, frac_white ≤ 0.20); **≥120 FPS median @1440p** (best of 2, per ADR-021); plus a **regression** pass that re-renders the M1/M2/M4 render goldens and asserts zero pixel diff.
- **Consequences:** Committed alongside this ADR (INV-8); measured at ~179 FPS @1440p with margin. Goldens change only via `goldgen` + this ADR (Iron Rule 6). The luminance band is intentionally wide (per-pose framing varies); it catches gross regressions (black screen, white-out), not subtle shading drift — that's what the bit-exact goldens are for.

## ADR-028 — M6 procedural audio + deterministic offline render (no miniaudio yet)
- **Status:** Accepted (M6).
- **Context:** The backrooms needs sound — a fluorescent mains hum, an HVAC bed, footsteps, room reverb — generated procedurally (no asset files) and **headless-verifiable** (INV-7). The audio must be a pure function of `(WorldSeed, input stream)` so the offline render is reproducible and gate-checkable (INV-1).
- **Decision:** A deterministic, block-based `audio::Synth` (`/`-free of wall-clock): 60 Hz hum + harmonics, lowpassed-noise HVAC, footstep transients, and a Freeverb-style reverb whose RT60 comes from the room probe (ADR-029). Footfalls are derived in **`core`** as a pure floor of the odometer (`footstep_count`, `kStrideLength`) and the listener via `audio_listener` — both return only `contracts::audio_events_v1` types, so `core` stays audio-free (INV-5). The app's `--render-wav` drives the deterministic maze walk (the walk-bot), renders exactly `sr/120 = 400` audio frames per sim tick (integer, no resampling), and writes a hand-rolled PCM16 WAV plus a footstep-tick log. **No `miniaudio` dependency is added in M6:** every M6 gate is headless and deterministic, so a device-playback library buys nothing gate-testable and would risk nondeterminism. `miniaudio` stays earmarked (ARCHITECTURE dependency ledger) for real-time speaker output, to be added with its own ADR + `vcpkg.json` entry (Iron Rule 8) when that ships.
- **Consequences:** The offline WAV is bit-identical across runs/processes on a fixed toolchain; footstep ticks reproduce from a replay with **no change to `replay_v1`** (they're derived, not stored). WAV I/O is header-only (`audio/wav.h`), shared with `wavcheck`. Real-time playback is deferred (headless-first), so M6 ships sound you verify by spectrum, not by ear.

## ADR-029 — M6 room-probe reverb (raycast mean-free-path -> RT60)
- **Status:** Accepted (M6).
- **Context:** Reverb should track the space the wanderer is in — a tight corridor vs an open hall — without authored room volumes, and deterministically.
- **Decision:** `audio::probe_reverb_seconds` casts 16 horizontal rays from the listener at the wall AABBs (slab method, XZ footprint), averages the nearest-hit distance into a **mean free path**, and maps it to an RT60 (`0.15 + mfp*0.05`, clamped `[0.20, 1.80]` s). The app gathers the surrounding chunk walls (the same 3×3 `GenerateChunk` gather as M4 collision, keeping `core` gen-free) and re-probes ~10×/s during the offline render. The reverb feeds the synth's comb feedback (`g = 10^(-3*delay/RT60)`).
- **Consequences:** Reverb is pure and varies with the maze; a unit test asserts a 25 m room rings longer than a 3 m room. The probe is 2-D (full-height walls) and ignores the ground box (inside-box rays don't count), which is exact for the backrooms' vertical walls; sloped/var-height geometry would need a 3-D probe later.

## ADR-030 — M6 audio gate: WAV spectrum + footstep alignment + real-time soak
- **Status:** Accepted (M6).
- **Context:** Audio needs machine-checkable gates with no speakers: prove the render is right (spectrum, not silence), the footsteps line up with the sim, and the real-time mixer never starves or stalls the sim.
- **Decision:** `gate.ps1 M6` checks: (1) **offline WAV deterministic** across two fresh processes (`Get-FileHash`) and its **spectrum** passes `wavcheck assert` — a self-contained radix-2 FFT finds the 60 Hz fundamental + 120/180 Hz harmonics each well above a 1.5–3 kHz noise floor, and RMS clears a silence threshold (the "WAV spectrum" golden of ARCHITECTURE §2, computed not stored — audio is per-toolchain like renders are per-GPU); (2) **footstep 1:1** — `--render-wav`'s audio footstep log is byte-identical to `--footsteps`' independent replay-derived log; (3) **real-time soak** — `--audiosoak --audio` runs the mixer thread (prebuffered producer, ~170 ms headroom, fed lock-free) for `-AudioSoakSeconds` with **zero underruns**, and the sim tick time with audio on stays within 1.5× of off (the audio thread never blocks the tick loop). Plus a render-golden regression (M6 touches no render code) and the standard isolation/inventory checks.
- **Consequences:** No committed audio byte-golden (machine-specific); robustness comes from determinism-x2 + spectral bands + the alignment cross-check. Measured: WAV bit-identical, 60 Hz at ~12000× floor, 64/64 footsteps aligned, 0 underruns over 60 s (and a 10-min soak), tick-time delta ~0.4%. The soak's underrun semantics model a device ring with headroom, not a literal driver.

## ADR-031 — M7 biome field + biome-parameterized generation (golden re-capture)
- **Status:** Accepted (M7, phase 1–2).
- **Context:** The world should vary — classic yellow, cubicle farm, pipe corridors, parking garage, poolrooms — in **contiguous regions** with controllable proportions, without breaking per-chunk purity (INV-2), cross-seam doorway agreement, or connectivity (INV-3).
- **Decision:** `gen/biome.{h,cpp}` assigns a biome per chunk by a pure, **low-frequency** function: a coarse **K=3** chunk lattice, each cell drawn from a weighted CDF (`biome_at`), giving ~96 m contiguous regions with designed weights (ClassicYellow 44 %, CubicleFarm 22 %, PipeCorridors 16 %, ParkingGarage 12 %, Poolrooms 6 %). Each biome supplies `BiomeParams` (interior **carve ratio** = openness, pillar density, floor tint, wall darken). `generate_layout_carve` takes the carve ratio (spanning tree first → carving only removes walls → **connectivity holds for any ratio**), and `GenerateChunk` applies the biome tint. The **edge-doorway protocol (`door_index`) is untouched**, so adjacent chunks of *different* biomes still agree on seam doorways (INV-3 across biome boundaries). Biomes do not change the per-chunk vertex *positions*, so seam-crack tests stay green.
- **Consequences:** `GenerateChunk` output changes (tint + wall density per biome), so the M4 top-down (seeds 1, 7) and M5 lit-shot (seeds 1, 7, 42 × 5 poses) goldens were **re-captured via `goldgen capture`** in this commit (Iron Rule 6) — a legitimate generation change, not a gate relaxation; determinism (bit-identical ×2) and the luminance band still hold. The distribution gate (102,400 chunks within ±2 % of the weights) and per-biome 10k-chunk connectivity pass. Pillars (parking garage / pillar halls) and the geometry-validator extension they need are deferred to the set-piece phase; verticality (level −1) follows.

## ADR-032 — M7 pillars (set-piece columns) + geometry-validator extension
- **Status:** Accepted (M7, phase 2b).
- **Context:** Parking garages and pillar halls need free-standing **columns**. The M4 geometry validator only accepted *walls* (thin in exactly one axis); a square pillar (thin in both) tripped it. Pillars must also never seal the maze (INV-3) or wedge the wanderer.
- **Decision:** `GenerateChunk` adds a 0.5 m square, full-height collidable **pillar** at a cell's centre with probability `BiomeParams::pillar_density`, consumed from the chunk RNG only when the biome calls for it (so pillar-free biomes stay bit-identical). A centred 0.5 m column in a 4 m cell leaves ~1.75 m clearance on every side — far more than the 0.7 m capsule — so cell-to-cell passage (what `validate_connectivity` flood-fills) is untouched, and pillars sit at cell centres where walls (cell edges) never reach, so no overlaps arise. `ValidateChunkGeometry` now classifies each collision box as a **wall** (thin in exactly one axis) or a **pillar** (thin in both, ≤ 1 m square); only a box thin in *neither* axis (a fat block) is rejected. The `--biomeat` app mode reports the biome at the spawn chunk so goldens/gates can target a known biome.
- **Consequences:** Pillar biomes' geometry now validates; the per-biome geometry test (2,000 chunks each) and the walk-bot (1 km × 5 seeds, 0 stuck) stay green with pillars collidable. Per-biome lit goldens `goldens/m7/biome_<name>.png` captured (parking garage shows columns); M5 seed-42 shots re-captured (its spawn is a pillar biome). Pillars are decoration + collision only — no new material slice (they reuse the baseboard texture).

## ADR-033 — M7 verticality: stacked levels + stairwell descent
- **Status:** Accepted (M7, phase 3).
- **Context:** The backrooms descend — a stairwell to **level −1** with altered generator params. This needs multi-level geometry, cross-level connectivity, and a deterministic, replayable descent, all without a multi-level streaming rewrite (deferred).
- **Decision:** Levels stack in world Y: `contracts::level_base_y(level) = level * kLevelHeight (4 m)`, so **level 0 maps to Y=0 (unchanged — no golden ripple)** and level −1 sits at Y=−4. `GenerateChunk` offsets all geometry by the level base Y and dims sublevels (×0.6 tint); `ValidateChunkGeometry`'s floor check is relative to the level base. `gen::build_stairwell` emits a **set piece** of 8 descending solid step boxes (0.5 m drop each) from level 0 to −1 — collision-only (rendering integrates with streaming later). The `app --descend` mode builds a level-0 approach floor + the stairwell + a level −1 landing and walks the wanderer forward: **gravity + the existing capsule-vs-AABB collision carry it down** (only step-*down* is needed, which the M2 collision already does). It reports the drop, level reached, sublevel connectivity/geometry, and the determinism hash.
- **Consequences:** Measured: a clean **4.019 m drop (exactly one level)** to level −1 across seeds 1–42, hash-reproducible (determinism), with the landing's level −1 chunk connected + geometry-valid. A unit test validates 4,000 level −1 chunks (connectivity + geometry, precision-safe coords per ADR-022). ## ADR-034 — M8 VHS post-process pass + HUD overlay
- **Status:** Accepted (M8).
- **Context:** The look needs the analog-horror "VHS" treatment — grain, chromatic aberration, lens warp, scanlines, vignette — plus a HUD/timestamp, all deterministic enough to golden and cheap enough to ship.
- **Decision:** A single **fullscreen post pass** in `render_d3d12`: the scene RT is bound as an SRV and a depth-less fullscreen-triangle PSO writes the composited result into a second `postRt` (read back instead of the scene). The PS applies **barrel distortion** to the scene sample coords, **chromatic aberration** (per-channel radial offset), **scanline + interlace flicker** and **vignette** in screen space, and **seeded film grain** (`hash(px,py,seed + floor(time*60))`, so a fixed time → fixed grain — deterministic goldens). Params ride in root 32-bit constants (`PostParams`). The **HUD** is rasterised on the **CPU** (a 5×7 bitmap font, `app/hud.*`) into a transparent RGBA overlay (TIME `HH:MM:SS` from the 120 Hz tick, seed, odometer, chunk, level, FPS), uploaded as a texture and composited **undistorted** (crisp) in the same pass. Post is **off by default** (`set_post`), so every prior golden is byte-unchanged. The timestamp is also echoed to telemetry for an OCR-free gate.
- **Consequences:** `gate.ps1 M8` checks: post ON/OFF goldens (`goldens/m8/`) bit-identical ×2 (seeded grain) + golden-matched + a clean A/B + debug-clean (#1, #4); the timestamp renders the right sim time via telemetry **and** a pixel golden (#2); and the post pass costs **< 1.5 ms at 1440p** — measured ~0.6 ms (#3). Plus a render-golden regression (post-off path unchanged). The HUD overlay is re-uploaded per frame only where values change (the perf path runs VHS-only); animated grain/interlace use the live time in real-time mode.

## ADR-035 — M9 DXR toolchain: runtime DXC + self-contained render_dxr
- **Status:** Accepted (M9, phase 1).
- **Context:** DXR raytracing shaders require **Shader Model 6.3+ DXIL**, which the FXC/`D3DCompile` path (SM 5.0) used everywhere else cannot emit. DXR also needs `ID3D12Device5` and a raytracing tier the device actually supports. `render_dxr` must obtain both without disturbing the raster renderer (INV-6: enhancement-only).
- **Decision:** **(1) Runtime DXC.** `render_dxr/dxc.*` loads `dxcompiler.dll` at runtime (`LoadLibraryW`, falling back to a scan of `…\Windows Kits\10\bin\10.*\x64\`) and compiles HLSL libraries to **signed** DXIL via `IDxcCompiler3` (the validator `dxil.dll` sits beside `dxcompiler.dll` and is found automatically — so signing works without copying anything). This is a **system dependency** (Windows SDK, like `d3dcompiler`/`d3d12`), **not** a vcpkg port — no `vcpkg.json` change. The FXC raster path is untouched. **(2) Self-contained `render_dxr`.** It creates its **own** `ID3D12Device5` (same HIGH_PERFORMANCE adapter selection as raster) rather than literally sharing `render_d3d12`'s device object — cleaner isolation + headless determinism; both renderers consume the same `contracts::ResidentChunk` geometry, so the cross-renderer depth compare (gate #1) is apples-to-apples.
- **Consequences:** `probe_caps()` (exposed via `app --dxr-probe`) verifies the whole toolchain end-to-end: measured **RTX 4070 Ti SUPER, Device5, RaytracingTier 1.1, DXC compiling a DXR library to signed DXIL** (2088 bytes). No new vcpkg dep. The probe degrades gracefully (reports `dxr_ready: 0`) on machines without DXR/DXC, so the gate can skip PT cleanly there. Reconciles `render_dxr` MODULE.md "device/share" → "same adapter-selection + same geometry contract".

## ADR-036 — M9 cross-renderer depth gate + path-traced lighting + converged golden
- **Status:** Accepted (M9, phases 2b + 3).
- **Context:** M9 needs two machine-checkable proofs: (gate #1) the DXR acceleration structures contain **exactly** the geometry the rasteriser draws, and (gate #2) a **converged**, **deterministic** path-traced image of the fluorescent-lit maze. Path tracing is stochastic and GPU float math reassociates across vendors, so a bit-exact cross-vendor golden is impossible; determinism must hold per-GPU.
- **Decision:** **(1) Depth compare (gate #1).** Both renderers emit **NDC depth** through the *same* hyperbolic `proj_lh(near=0.05, far=500)` mapping. The DXR ray basis (`fwd = (sin·cos, sin, cos·cos)`, `right = cross((0,1,0),fwd)`, `up = cross(fwd,right)`) already equals raster's `view_lh`, and the screen mapping matches, so per-pixel primary rays align with **no** basis change. `render_dxr` writes depth from the hit's eye-space z; `render_d3d12::readback_depth` copies the D32 buffer. `app --dxr-depth` linearizes both NDC buffers to eye-space metres (NDC is non-linear) and compares with a relative epsilon, classifying pixels as co-foreground / co-background / silhouette-edge. **(2) Path tracer (gate #2).** Inline DXR 1.1 **`RayQuery`** (SM 6.5, compiled via the same runtime DXC at `lib_6_5`) in a raygen-only state object — no hit groups/recursion. All resident chunk vertices are concatenated into one `StructuredBuffer` and each TLAS instance tags its **start vertex offset in `InstanceID`**, so the shader reads per-hit normal/material via `(InstanceID + 3·PrimitiveIndex)`. Lighting = the emissive fluorescent ceiling grid as area lights (analytic **NEE + shadow rays** over the nearby grid cells, computed from the `is_fluorescent_cell` formula — no light list), one cosine-weighted **diffuse-GI bounce**, plus a small **ambient interreflection floor** (1-bounce GI underestimates an enclosed room). Per-(pixel,sample) seeded **PCG hash** RNG; radiance accumulated in an **RGBA32F buffer across batched dispatches** (≤64 spp/dispatch, under the GPU watchdog), then Reinhard-tonemapped on resolve. **(3) Converged golden.** Render **1024 spp** at 3 fixed poses (`goldens/m9/pt_pose{1,3,4}.png`, captured by `goldgen`); the gate re-renders and compares by **mean-abs channel diff < 1.0** (RMSE-style, not bit-exact) to absorb cross-vendor reassociation, plus a **determinism** check (two renders bit-identical on the dev GPU), a **luma band**, and **debug/DRED-clean**.
- **Consequences:** `gate.ps1 M9` gate #1 = 5-pose depth compare (measured: every pixel co-foreground, mean depth rel-err ~1e-5, ≤1 silhouette pixel); gate #2 = 1024-spp PT at 3 poses (measured: deterministic, ~1.06 s/pose on the dev GPU, diff-vs-golden 0.0, luma in band, debug-clean). Raster stays the default + fallback (INV-6); `render_scene` (recursive, gate #1) and the inline `render_pt` (gate #2) coexist over the same BLAS/TLAS. Goldens are bit-exact on the dev GPU; the RMSE threshold is the documented cross-vendor allowance.

## ADR-037 — M10 soak harness: duration-parameterized soak + minidump + contact sheet
- **Status:** Accepted (M10).
- **Context:** M10 must prove long-haul stability — an **8 h** headless soak with zero crashes, flat memory, frames that keep rendering above a floor, and connectivity that holds — plus crash forensics (minidump → auto-restart). ## ADR-038 — M11 Director inference via the KEEL sidecar (OpenAI HTTP), not embedded llama.cpp
- **Status:** Accepted (M11).

## ADR-039 — M11 Gate 2 reformulation: async-isolation metric corrected for shared-GPU inference
- **Status:** Accepted (M11). *The invariant is unchanged; only the metric is corrected — recorded here because a gate redefinition must be a reasoned, deliberate act, not a quiet wording change.*
- **Context:** M11's original Gate 2 read *"Frame-time telemetry identical (± noise) with generation running vs. idle (proves async isolation)."* That metric was written for the **embedded-llama.cpp** plan. Routing the Director through the **KEEL sidecar** (ADR-038) surfaced a physical truth the embedded plan would have hidden: KEEL's local **Qwen-9B inference and the D3D12 renderer share one GPU**, so live generation shifts the *render* frame-time baseline (measured 6.9 ms vs 3.9 ms median under an aggressive 2 s submit cadence). That is **GPU resource contention, not a sim-loop stall** — diagnosed cleanly: with the Director worker active but inference unreachable (dead URL → no GPU work), frame-time returns to baseline (4.1 vs 3.9 ms), and the hitch profile (p99/median ≈ 1.0×) is unchanged ON vs OFF. The async-isolation **invariant** (the Director never blocks/stalls the sim/frame thread) is intact; the literal "frame-time identical" **proxy** is confounded by shared-GPU physics.
- **Decision:** Keep the invariant; correct the metric. Gate 2 = **async isolation**, measured as **A + B** (skip a GPU-independent sim-tick check — Gate 4 already proves the sim is Director-independent, and render smoothness is what a player actually feels):
  - **A.1 — integration overhead ≈ 0:** Director worker active but inference unreachable (dead URL) → soak median ≈ OFF. Proves the thread + non-blocking `submit`/`poll` + event-log apply cost ~nothing on the main thread.
  - **A.2 — no new hitches:** under live inference, the hitch ratio (p99/median) stays within a small factor of OFF (a blocking Director would explode it).
  - **B — lived smoothness:** under realistic **ambient** pacing (a summary every ~15 s wall — the rate a player experiences, not every 2 s), sustained soak median stays within a **stated tolerance band** of OFF. It's true *because* the Director is ambient, not because of throttling-to-game.
  - The instantaneous live-inference shift is **documented as expected shared-GPU contention** (the sovereign-local tradeoff), never claimed as "identical."
- **Consequences:** `gate.ps1 M11` Gate 2 runs three short soaks (off / dead-url / live-ambient, the live one exercising ≥1 real inference) and asserts A.1 (deadurl median ≤ off×1.4), B (live median ≤ off×1.6), A.2 (live p99/median ≤ off p99/median ×1.6). **Measured (30 s, 15 s pacing): deadurl ~1.05× off, live median ~1.07× off, live p99/median ~1.03× off** — all far inside. Routing through a real substrate surfaced a truth worth recording.

## ADR-040 — M14 real-time audio output via miniaudio (header-only), gated on the null backend
- **Status:** Accepted (M14).
- **Context:** M6 built the entire procedural audio stack — deterministic `Synth`, room-probe reverb, offline `--render-wav`, and a headless real-time `AudioEngine` that renders blocks against a *virtual* wall-clock read cursor (the M6 comment: "exactly as a device callback + ring would"). Master + SFX volume are presentation-only atomics applied at mix time (default 1.0 → M6 behavior byte-for-byte). **`--play` opens the default device** (silent no-op if none); **`--audiodev [--null]`** is the headless gate mode.
- **Gate uses the null backend.** miniaudio ships a hardware-free **null device** that pumps the data callback at real-time cadence with no speakers — so the M14 gate (`--audiodev --null`) exercises the *exact* open→callback→ring-drain→close path **deterministically and on any host (CI included)**, consistent with the project's headless-first doctrine. The real WASAPI path is smoke-verified on the dev box (`--audiodev` → `backend: WASAPI`, 0 underruns) and is what a player hears in `--play`; it is not a hard pass/fail (a headless host may have no default endpoint).

## ADR-044 — M18 realistic walking: head-bob as a view-only function of the deterministic odometer + a run modifier

## ADR-045 — M19 real-time ray-tracing toggle: DXR-headless → upscaled present, default off (no regression)
- **Status:** Accepted (M19). *Records the re-captured `goldens/m15/settings.png` (Iron Rule 6: via `goldgen` + this entry).*
- **Context:** the game had a DXR path tracer (M9) but only as a capture mode (`--dxr-pt`); the live walk was always raster. The operator wants ray tracing as an **in-game toggle**, **default off so it can't regress** anything. The obstacle: `render_dxr` is a **separate `ID3D12Device5`** that does headless readback, while `render_d3d12` owns the swapchain — two devices can't trivially share a swapchain.
- **Decision — sidestep cross-device sharing: DXR renders headless, its frame is *presented* by the raster renderer.** When the toggle is on, the live loop renders the path tracer at a **reduced internal resolution** (2/3 of the window, for perf), `readback`s the RGBA, and hands it to `render_d3d12::present_overlay_windowed`, which now **sizes its overlay texture to the source** and the fullscreen-triangle sampler **upscales** it to the window. Accumulation resets when the camera moves (fresh few-spp frame) and converges when still; the scene rebuilds when the chunk center changes. The toggle is a **Settings row** ("RAY TRACING", `kSettingsItems` 5→6), an **F2** hotkey, a `--rt` flag, and it **persists** in `backrooms.cfg` (the long-reserved `renderer` field). `--play`/`--game` report `rt_frames` + `rt_luma_mean` for the gate.
- **Consequences:** **Default off = raster = zero regression** — `gate.ps1 M19` proves the **M5 raster golden is bit-identical** and that `--play` (no `--rt`) runs `rt_frames=0`. RT-on is gate-checked live: `--play --rt` traces **every frame** (663 DXR frames), debug-clean, luma in band. No new dependency (it's the existing `render_dxr`). The **settings menu golden was re-captured** for the new row (only `settings.png` changed; splash/mainmenu/pause are untouched). The DXR-headless→present bridge is reusable for **M22's shoggoth-vision snapshot**. **Next: M20** the Shoggoth's body + deterministic movement.

## ADR-048 — M21b the Shoggoth's LIVE async brain: KEEL off the frame thread, so it thinks while you play
- **Status:** Accepted (M21b). *Completes the M21 brain — the cascade now runs in the live game, not only in the headless record/replay gate.*
- **Context:** M21 gave the Shoggoth a KEEL brain, but it ran only in the headless `--shoggoth-record` / `--shoggoth-replay` modes (which prove the sacred determinism gate). In the actual playable game (`--game` / `--play`) the creature ran with the default `Hunt` intent — it never thought live. M21b makes it think WHILE you play, without ever hitching the 120 Hz frame loop. A blocking KEEL call (multi-second local inference) on the frame thread would freeze the game; the entire M11/Gate-2 lesson (ADR-039) is that AI generation must live behind an async boundary.
- **Decision — mirror the Director's async host.** `app/shoggoth_brain_host.h` is a header-only `ShoggothBrainHost` that is a faithful mirror of `director::DirectorHost`: a background worker thread, a non-blocking latest-wins `submit(ShoggothSummary)` (≤1 inference in flight, always reasoning about the present), a non-blocking `poll()` that drains validated intents, and atomic `requests()`/`produced()` counters. The worker runs the **exact M21 brain** off the frame thread — `keel_complete` → `render_shoggoth_prompt` → `parse_shoggoth_intent` — so KEEL latency never enters frame time. Wired into `run_play` + `run_game`: every ~3 s (ambient wall-clock) the loop submits a fresh summary and applies any returned intent to `sh.intent` **at a tick boundary** (the same "intent is a discrete, timestamped event" shape the headless sacred gate proves bit-exact — the LLM never couples continuously to the sim). On by default; **`--no-shoggoth-brain`** is the kill switch (INV-6); a graceful no-op when KEEL is down (failed connect → `ok=false` → nothing applied → the creature keeps hunting on the M20 default).
- **Determinism is untouched.** M21b changes only the LIVE presentation path; the headless `--shoggoth-record`/`--shoggoth-replay` determinism is byte-for-byte unchanged (gate-checked: 5 LLM intents, record == replay `12e9f56f9171fe5d` with the model offline). Interactive play is inherently non-replayable (it records no input stream), so no event log is written there — the determinism PROOF stays the headless gate. *(Implementation gotcha: `parse_host_port` lives in a nested anonymous namespace in `main.cpp`; a plain forward declaration for the earlier `run_play`/`run_game` created a second overload → `ambiguous call`. Fix: wrap the forward decl in `namespace { … }` so it lands in the same anonymous namespace as the definition.)*
- **Consequences:** `gate.ps1 M21b`: clean build + full ctest (incl. a `[m21b]` host-lifecycle test) + KEEL reachable + **the live brain thinks (≥1 intent applied in `--play`) AND stays async-isolated** + graceful dead-URL no-op + the `--no-shoggoth-brain` kill switch + `--game` debug-clean + the M21 sacred gate + M20/M5/INV-5/inventory regression. **Async isolation is proven two ways** (the M3 + M11/Gate-2 lessons): (a) absolute — p99 frame < 2× median, best-of-2 at 1440p (a blocking host would explode this to ~100×); (b) relative — the brain-ON hitch ratio adds nothing over brain-OFF. **Measured (8 s `--play`, 1440p): brain-on p99/median 1.79× vs brain-off 1.44× — ON ≈ OFF, the async host adds no frame-thread cost** (the residual is the uncapped loop's shared-GPU jitter, ADR-039, not a stall); the live brain applied 3 intents over 8 s. No new dependency (it reuses the Director's KEEL client). The monster now thinks while you play. **Next: M22** vision (a POV snapshot → qwen-VL via a backrooms-local KEEL copy) feeds richer intent to this same brain — operator-gated (stop before M22).

## ADR-051 — M24 the Backrooms PA gets a VOICE: a from-scratch procedural formant TTS whisper can read back
- **Status:** Accepted (M24). *The richer hearing loop M23 deferred — the monster hears actual WORDS, not just ambient tags.*
- **Context:** M23 gave the Shoggoth hearing, but the Backrooms is non-speech, so whisper only ever returned coarse tags (`(upbeat music)`). The operator asked for the richer loop: speak the Backrooms PA so whisper recovers real words. The constraint is the project's hardest rule — **no asset files**, everything procedural — which rules out any recorded/concatenative/diphone TTS. So the voice must be **synthesized from first principles**.
- **Decision — a Klatt/eSpeak-style formant synthesizer, header-only and pure.** `app/tts.h` turns text → phonemes (a Backrooms-PA lexicon + a letter-to-sound fallback) → formant tracks → 16-bit PCM, with no assets and no dependency. The result is intelligible and fittingly inhuman. `--tts-say` writes a line to a WAV; `--tts-check` round-trips text → TTS → whisper and reports the words recovered.
- **The PA-voice hearing loop.** `run_shoggoth_pa_record` (`--shoggoth-pa-record`) renders the soundscape at the Shoggoth's ears with a spoken Backrooms PA announcement (the TTS) mixed over a quiet ambient bed, runs whisper (default `large-v3-turbo` — speech wants the stronger model), and feeds the heard line to the brain. One determinism bug fixed: a `static thread_local` lowpass state in the synth leaked between calls — made a per-call local.
- **Consequences:** `gate.ps1 M24`: clean build + full ctest (incl. `[m24]` TTS determinism + lexicon tests) + whisper (both models) + KEEL reachable + **the TTS is intelligible** (`--tts-check "EVACUATE SECTOR FIVE"` → whisper recovers ≥2 spoken words — measured "Evacuate sector 5", 2/3) + **THE SACRED GATE, PA-VOICE EDITION** (spoken PA → whisper words → intent → record == replay all-offline) + two graceful no-ops + the M21 text sacred gate + M20/M5/INV-5/inventory regression. **Measured: whisper reads the procedural voice cleanly** ("Evacuate sector 5", "Level 3 containment breach"); 5 PA intents, record == replay. The Backrooms now *speaks*, and the engine stays bit-exact. **Next: M25** — the Shoggoth body in the ray-traced (DXR) path.

## ADR-052 — M25 the Shoggoth's body in the ray-traced path: a dynamic creature BLAS (chunk BLASes cached)
- **Status:** Accepted (M25). *Completes the in-world body — visible in BOTH renderers now; the Shoggoth arc is whole.*
- **Context:** M20b put the Shoggoth's procedural body in the world, but only in the RASTER path (injected as a synthetic `ResidentChunk`). M25 makes the creature appear + writhe in RT too, without that cost.
- **Decision — a dynamic creature BLAS, with the chunk BLASes cached.** `DxrRenderer::update_creature(verts, count)` updates a single creature mesh per frame WITHOUT rebuilding the chunk acceleration structures. `build_scene` now reserves a tail of `kMaxCreatureVerts` (4096) in the shade buffer and caches the chunk TLAS instance descriptors + the chunk vertex count; `update_creature` writes the creature's verts into that reserved tail (so the PT reads them via `InstanceID = chunkVertCount`), builds ONE small creature BLAS, and rebuilds ONLY the TLAS (cached chunk instances + the creature instance). Per-frame cost is one BLAS + one TLAS build, not 169 — cheap enough to animate the creature every frame. The PT shades by **material id**, so the creature is tagged **material 7** at injection (a salmon `albedo_of` branch in the scene shader); the raster path is untouched (it shades by vertex colour, so M20b/M5 are byte-for-byte identical — the material override is DXR-only).

- **Status:** Accepted (operator-reported bug). A **safety net** re-picks immediately if a wall ever looms within 1.5 m of the committed heading; a no-progress counter frees a rare wedge. The camera turns at a calm clamped rate and **eases forward speed off while pivoting** (it rounds the corner instead of scraping it), with a subtle idle weave so it does not feel rigid. `WalkBot` is untouched; the `Stroller` is screensaver-only and cannot perturb any gate.

## ADR-061 — screensaver: an ORGANIC head-bob (a human gait, not a perfect sine)
- **Status:** Accepted (operator-reported). With the Stroller's smooth walk in place, the operator asked the camera to "realistically bob up and down — a human walking isn't perfectly level, like a sine wave but not a *perfect* sine wave."
- **Context:** the shared M18 `head_bob` is a clean cosine/sine (two dips per stride + a figure-8 sway), driven by the deterministic odometer. It is correct but *metronomic* — a perfect repeating wave, which is exactly what the operator did not want. M18 is also shared by `--play`/`--game` and pinned by unit tests (`dy <= 0`, two-dips-per-cycle), so it must stay as-is.
- **Decision — a screensaver-only `apply_organic_bob`** (view-only, like M18: driven purely by the deterministic odometer/speed, never touches `WorldState`). On top of the two-dips-per-stride base it layers what makes a real gait *not* a pure sine: (1) **left/right asymmetry** — alternating footfalls land at slightly different depths (a subtle dominant-leg "limp", ~13%), (2) a **2nd harmonic** so the dip shape isn't a pure cosine, (3) three **slow, incommensurate modulators** (periods ~48 m / ~94 m / ~126 m of walking) that drift the vigor, the asymmetry, and a not-quite-level lean independently → the motion is **quasi-periodic and never exactly repeats**, and (4) a small **pitch nod** (~1.5° range) coupled to the steps + the slow lean, so the head nods and is never perfectly level. The vertical term stays `<= 0` (the head only dips). Amplitude ~6 cm walk (a touch deeper than M18's 5 cm — clearly felt for a passive screensaver). The crisp M18 bob is unchanged for the game.
- **Consequences:** `--strollcheck` now also reports the bob: `bob_amp_cm`, `dip_spread_cm` (the footfall dips VARY → organic), and `nod_deg`. Measured (5 seeds): **bob ~7.4 cm peak**, **dip spread ~5.3 cm** (dips are clearly non-uniform), **nod ~1.5°**. The M30 gate's `stroll_ok` now also requires `bob_amp > 3 cm` and `dip_spread > 0.8 cm`, so the bob can't regress to flat or to a perfect repeat. Full ctest 100/100; cosmetic/non-gated; no new dependency.

## ADR-062 — screensaver nav v2: GOAL-DIRECTED, FREE-ANGLE traversal (supersedes the cardinal Stroller of ADR-060)
- **Status:** Accepted (operator-reported). The cardinal Stroller (ADR-060) fixed the WalkBot faceplanting but the operator found it still "like a robot vacuum": it moved on a **cardinal grid** — only 90° turns (1995-DOOM, not how humans turn), cell-by-cell, and it **hugged corridor centre-lines / walls even in open rooms** instead of cutting across. It felt aimless (vacuuming a space), not like a person doing a real traversal.
- **Context:** the three faults are all consequences of grid-locked navigation: (1) headings snapped to {0,90,180,270}; (2) movement was cell-to-cell with no destination, so it wandered; (3) it followed the maze grid, so in an open room it traced the walls rather than the diagonal a person would take.
- **Decision — replace the grid walker with a GOAL + CONTEXT-STEERING navigator** (the standard game-AI steering model; still view-only, never touches `WorldState`). (1) **Goal:** it picks a distant destination — scan the circle for the longest open runs, favour the longest + mostly-forward, plant a goal near the end of that run (across a room or down a hall). On arrival (or if a dead-end stalls it for ~3.5 s) it picks a new one → **real traversals**, not vacuuming. (2) **Free-angle steering:** a fan of 24 look-ahead feelers each returns its clearance; the chosen direction maximizes `1.7·cos(toGoal) + 0.9·(clearance/reach) + 0.5·cos(toFacing)` — i.e. head toward the goal, keep margin from walls, don't whip around. The result is a **continuous angle**, and the camera chases it at a clamped rate → smooth, **no 90° snapping**. (3) **Open-room cut-through:** the clearance term is equal in all directions in an open room, so the goal term wins → it beelines diagonally to the goal instead of hugging walls; near a wall the clearance term steers it to the open lane (margin, no hugging); in a corridor only the along-hall directions are clear, so it follows the hall centred. Holes/stairs read as blockers (it stays on its floor).
- **Consequences:** `--strollcheck` adds `offcardinal_deg` — the mean angular distance of the facing from the nearest cardinal (a 90°-locked vacuum ≈ 0°; a free-angle human is well off zero). Measured (5 seeds): **offcardinal 13.6–22.7°** (uniform-random is 22.5°, the old cardinal nav ≈ 0–3°), **distance ~1110–1140 m**, span 50–131 m, **faceplant 0.011–0.045** (a touch higher than the cardinal walker because free-angle paths legitimately approach a wall before steering around it — still ~95–99% looking where it is going; WalkBot was 0.94). The M30 gate's `stroll_ok` now also requires `offcardinal_deg > 6` (not 90°-locked) and the faceplant ceiling is loosened 0.05 → 0.10/0.12. Full ctest 100/100; cosmetic/non-gated; no new dependency.

## ADR-063 — screensaver nav v3: X-RAY BFS PATHFINDING + look-ahead following (supersedes the steering-only v2)
- **Status:** Accepted (operator-reported). The v2 free-angle steerer (ADR-062) still felt like a vacuum: with only ~6.5 m of local feelers it would commit toward a blind goal, walk into a concave dead-end it could not see, and back out -- the operator's "back-and-forth." They asked it to "cheat" with X-ray vision: pre-compute a path so it never reaches a dead-end and always knows where it is going.
- **Decision -- a GLOBAL planner + LOCAL steerer.** (1) **X-ray BFS** (`Stroller::plan`) floods the DETERMINISTIC maze graph (`app::maze_open`, the same generator the world is built from -- the cheat) over a ~104 m window from the current cell, excluding floor-hole/stair cells, and picks a **far, STRONGLY-forward** goal (`dist × max(0, cos(rel))` -- never targets behind), reconstructing the cell path. The goal is reachable by construction, so **following the path can never hit a dead-end.** (2) **Following:** a look-ahead lead ~9 m along the path (around corners) feeds the v2 24-feeler steering fan, which picks the actual heading -- so it follows the guaranteed route but flows down corridor centres + around pillars (low faceplant), at a free continuous angle. It re-plans only on **reaching** the goal, committing to each route.
- **Consequences (two follow-a-path-smoothly bugs found + fixed):** (a) naive pure-pursuit *cut corners across walls* and pinned -> feed the lead to the fan, never aim straight at it; (b) advancing `path_i` by *proximity* to waypoint centres let the index **stall behind** the corner-cutting walker, so the look-ahead pointed backward and it ping-ponged -> advance by **progress** (step to the next waypoint whenever it is nearer). `--strollcheck` adds `stall_frac` (fraction of ticks whose position 5 s ago is < 4 m away = ping-ponging) + `uturns`. Before the fixes stall_frac was **0.42–0.68** (real back-and-forth); after: **distance ~970–1120 m, stall_frac 0.10–0.35, faceplant 0.07–0.22, offcardinal ~20°**. The M30 gate guards `stall_frac < 0.45` + `faceplant < 0.30`. Full ctest 100/100; reuses the Shoggoth's proven maze BFS; cosmetic/non-gated; no new dependency.

## ADR-064 — a double-click launches the GAME (no-arg default) + best-effort LLM sidecar auto-start
- **Status:** Accepted (operator-reported). Double-clicking `backrooms.exe` "always crapped out a fraction of a second after launch."
- **Context:** with **no mode flag** (exactly what a double-click passes), `main` fell through to a two-line `printf("Backrooms Sim v…")` + `return 0` -- so it flashed a console and exited instantly, which reads as an immediate crash. The game only ever ran via an explicit `--game` (or the menu), which a double-clicking user never types. Separately, the LLM only went live if the operator had manually started the KEEL sidecar.

(Explicit modes, `--version`, and the `.scr` `/s//p//c` detection are unchanged; CI/tests always pass flags.) (2) `run_game` calls a best-effort `try_start_sidecar()` when the creature brain is enabled: fire-once, it `CreateProcessW`-launches `C:\keel-sidecar-7071\start.cmd` hidden + detached **only if that launcher exists** (so it stays portable -- absent elsewhere = skip). Silent + non-blocking; the brain host (WinHTTP, per-request) reconnects on its own once the sidecar answers, and if the model backend is down it's the usual graceful no-op (deterministic AI). No reachability pre-check -- a duplicate launch self-terminates on the port bind.

The stale `build-release\bin\backrooms.exe` (a separate Release config, 6/14, broken for command-line rebuild -- its toolchain env is gone) was overwritten with the current working build, and a Desktop `Backrooms.exe` + refreshed `Backrooms_v2.scr` delivered. ctest 100/100. No new dependency (CreateProcessW/GetFileAttributesW are kernel32, already linked); determinism untouched (the sidecar only feeds the sim via recorded events).

## ADR-065 — DPI-awareness fixes the game's spinning camera on scaled displays + a robust mouse-look
- **Status:** Accepted (operator-reported, critical). On New Game the view "spun in circles ~100×/sec, unplayable -- couldn't see or walk."
- **Context:** the process was **not DPI-aware**. The windowed game sizes its swapchain from `EnumDisplaySettings` (PHYSICAL pixels, e.g. 1920×1080) and recenters the cursor in client→screen coords, but on a SCALED display (125/150/175%) a non-DPI-aware process gets the cursor virtualised to LOGICAL pixels. So the mouse-look recenter target and the `GetCursorPos` read-back live in different coordinate spaces -> `cur − anchor` is a large CONSTANT every frame -> the camera spins continuously. The screensaver dodged it by using `GetSystemMetrics` (logical) consistently; the game mixed physical (resolution) and logical (cursor). It only surfaced now because the no-arg double-click fix (ADR-064) finally let a user *reach* Play.
- **Decision.** (1) `SetProcessDpiAwarenessContext(PER_MONITOR_AWARE_V2)` at `main` start -> physical pixels everywhere; the cursor, the window, `EnumDisplaySettings`, and the swapchain all agree (also renders at true native res, sharper). (2) Defence-in-depth on `run_game`'s mouse-look so a stray delta can never runaway-spin again: **recenter then re-read** where the cursor actually landed (robust to a clamped/failed recenter -> no constant-offset spin), **skip the first Play frame** (the entry recenter jump), and **clamp** the per-frame look to ±0.5 rad.
- **Consequences:** process-wide DPI-awareness benefits every mode (game/play/screensaver -- all coord-consistent now). The game builds + launches clean (`--seconds 3`: 432 frames, debug-clean, exit 0); ctest 100/100. The spin fix itself is interactive (a real window + mouse on a scaled display), so it needs the operator's visual confirmation -- delivered to `build-release\bin` + Desktop. No new dependency (`SetProcessDpiAwarenessContext` is user32). Determinism untouched (view-layer only).
- **Superseded by ADR-066** as the *root-cause* fix: the DPI-awareness here is RETAINED (still correct + sharper native render), but it was NOT sufficient -- the spin persisted. A headless spin guard built next caught that.

## ADR-066 — raw-input mouse-look (the ACTUAL spin fix); a headless `--auto-play` spin guard that disproved ADR-065
- **Status:** Accepted (operator-reported, critical -- supersedes ADR-065's mouse-look mechanism; keeps its DPI-awareness).
- **Context.** ADR-065 theorised the spin was purely DPI coordinate-space mismatch and shipped DPI-awareness + a "recenter then re-read the anchor" scheme, *unverified* (the bug is interactive). Rather than ask the operator to re-test, I added a **headless spin guard** -- `--game --auto-play`: drop straight into Play with NO mouse input and measure the per-frame look yaw, which MUST be ~0 for a still mouse. It ran on the real display and **FAILED**: `lookcheck_max_dyaw = 0.50000` (pinned at the clamp) every frame. Instrumenting the coords exposed the true mechanism: `recenter()` set the cursor to the client centre and `GetCursorPos` read it back correctly, but at the **start of the next frame** `GetCursorPos` returned **(0,0)** -- the cursor was being warped to the top-left every frame, so `cur − anchor` was a constant ~(−1149,−920) -> clamped spin. The entire **absolute-cursor** scheme (GetCursorPos/SetCursorPos/recenter/anchor) is fragile: it breaks under cursor warping (a non-interactive desktop), DPI virtualisation (ADR-065's case), edge-clamping, and focus changes. ADR-065 treated one symptom of a broken design.
- **Decision.** Replace the absolute-cursor look with **raw input (`WM_INPUT`)** -- exactly how real FPS games read the mouse. `RegisterRawInputDevices` (usage page 1, usage 2) on the window; accumulate the RELATIVE `RAWMOUSE.lLastX/Y` per frame from the message pump; look yaw/pitch = `rawDelta * kSens`. Look NO LONGER depends on cursor position, so it is immune to DPI, screen edges, warping, and the (0,0) artifact; a still mouse emits no `WM_INPUT` -> zero delta -> the view physically cannot self-spin. The cursor is hidden + `ClipCursor`-confined to the window and parked at centre (cosmetic only). Applied to all three windowed look sites: `run_game`, `run_play`, and the screensaver's SPACE-play (`scr_proc` accumulates into a file-static, mirroring its existing `g_scrQuit`/`g_scrPlay` channel). Absolute-coordinate devices (RDP/tablet/touch) are skipped, not misread.
- **Permanent guard.** The `--auto-play` lookcheck is now an **M30 gate assertion** ("game mouse-look does not self-spin"): a still mouse over ~160 Play frames must give `lookcheck_max_dyaw < 0.05`; FAIL returns exit 4. The old scheme pinned it at 0.5; raw input gives **0.00000**. This is why headless-first (Iron Rule 3) matters: the guard disproved a plausible-but-wrong fix before it reached the operator.
- **Consequences:** spin verifiably dead on the real display (0 rad/frame, debug-clean, 100% gate + ctest 100/100). No new dependency (raw input is user32). Determinism untouched (windowed view-layer only; sim/replay/goldens never read the mouse). Fresh exe re-delivered to `build-release\bin` + Desktop `Backrooms.exe`/`Backrooms_v2.scr`. Mouse sensitivity reuses the existing `kSens` slider (raw counts ≈ pixel deltas in magnitude; the ±0.5 rad clamp stays as a spike guard).
- **Incomplete — see ADR-067:** this commit left a vestigial per-frame `recenter()` (a `SetCursorPos` "park") in the Play loop. Harmless for the *spin* (raw input ignores cursor position) but it warped the user's cursor every frame -> the "my cursor fights me" bug ADR-067 fixes.

## ADR-067 — stop the per-frame cursor warp ("my cursor fights me"); focus-gated capture + a cursor-warp guard
- **Status:** Accepted (operator-reported, critical -- completes ADR-066's raw-input switch).
- **Context.** After ADR-066 the operator reported their **mouse cursor was "fighting" them -- "no other game does this."** Two independent review agents (a codebase cursor-API sweep + an adversarial Win32 reviewer) confirmed the cause in seconds: ADR-066 switched the *look* to raw input but left the OLD per-frame `recenter()` (`SetCursorPos(window_centre)`) in the Play loop of `run_game`, `run_play`, AND the screensaver SPACE-play. With raw input that call is **vestigial** (look no longer reads cursor position) but it drags the OS cursor to centre ~100x/s. The decisive detail: in `run_game` the per-frame warp fired whenever `screen == Play` **regardless of window focus** -- so alt-tabbing or clicking away while in-game left the loop yanking the cursor back to the game-window centre every frame, fighting the user on their desktop. (`SetCursorPos` is the *only* cursor-moving call in the codebase; `ClipCursor`/`ShowCursor` never move the pointer. Sweep confirmed no other warp sites.)
- **Decision.** (1) **Delete the per-frame `recenter()`** from all three Play loops -- raw input needs no centring. (2) **Focus-gate capture** in `run_game`: `wantCapture = inPlay && (GetForegroundWindow()==hwnd)`. On gaining capture: `ShowCursor(FALSE)` + one-time park + `ClipCursor`(window); on losing it (alt-tab): `ShowCursor(TRUE)` + `ClipCursor(nullptr)` -- the cursor is **free the instant focus drops**, never trapped. `ClipCursor` is re-asserted each captured frame (Windows drops it on focus/mode changes) but never *moves* the cursor, so it cannot fight. Hide/show are strictly **edge-paired** (ShowCursor is a signed counter -- unbalanced calls strand the cursor). (3) **Focus-gate Play input too**: `GetAsyncKeyState` ignores focus, so WASD/jump/run/look and the F11/F2 toggles are now gated on foreground -- the wanderer no longer walks (and fullscreen no longer toggles) while you are alt-tabbed away, like a normal game. The screensaver SPACE-play swaps its per-frame warp for a one-time park + `ClipCursor` confine (it already exits on focus loss).
- **Permanent guard.** Added a process-wide `g_cursorWarps` counter bumped on every `SetCursorPos`. The `--auto-play` check now reports `lookcheck_cursor_warps` over the Play session and the **M30 gate fails if it is >= 10** (renamed "no self-spin AND no cursor-fight"). Result: **0 warps** over ~160 still-mouse frames (the vestigial per-frame warp would have been ~160). So the regression is now machine-checkable, not eyeballed.
- **Consequences:** the cursor-fight is gone by direct measurement (0 per-frame warps, was ~160), gate M30 green, ctest 100/100, debug-clean. The capture path is analytically airtight: it moves the cursor exactly **once** per capture (a hidden park), never per-frame; matches SDL relative-mouse-mode. Determinism untouched (windowed view-layer). Fresh exe re-delivered to `build-release\bin` + Desktop. The one thing still only verifiable by the operator is *feel* (sensitivity) and the interactive alt-tab release; both are standard and adversarially reviewed.

## ADR-068 — fix the noisy ray-traced mode: a procedural edge-aware denoiser (NOT DLSS) + a per-frame seed
- **Status:** Accepted (operator-reported: "the ray trace mode is horribly static noisy and unoptimized"; operator picked the procedural path over real DLSS).
- **Context.** Three investigation agents (forensics on the operator's Buddhabrot projects · a diagnosis of the Backrooms DXR path · a research sweep). Findings: (1) the operator never actually had **DLSS** -- the thing they got working before was the **OptiX AI Denoiser** (CUDA, OpenGL) in `C:\Buddhabrot_CUDA`. (2) The Backrooms DXR noise had **two real bugs, not a missing AI upscaler**: the interactive path passed `reset=true` **every frame** (so the accumulator was wiped to 4-spp-from-scratch each frame -- temporal integration plumbed but disabled), and the per-pixel RNG seed had **no temporal axis** (a fixed function of pixel+sample), so the grain was frozen. (3) Real **DLSS Ray Reconstruction** = best quality but ships ~40-90 MB of proprietary NVIDIA DLLs (which *are* asset files, violating the project's no-assets iron rule) + RTX-only + days of G-buffer plumbing; **from-scratch SVGF** = ~90% of the quality, zero dependencies, any GPU. Operator chose procedural.
- **Decision.** Add a **single-pass, multi-scale, edge-aware spatial denoiser** to the existing PT pipeline -- the spatial core of SVGF, no new pipeline/SBT/compute-PSO, no external deps. (a) A new **guide G-buffer** (u3, RGBA32F = world normal.xyz + view-Z) is written in the PT resolve from values the shader already computes. (b) A new shader mode (`uResolve==3`) blurs the accumulated radiance in **linear** space across à-trous dilations {1,2,4,8} px in one dispatch, stopping the blur at **depth + normal + luminance** edges, then tonemaps -> color. (c) `render_pt_frame` gains `denoise`/`frame` params: interactive paths (`run_game`, play/screensaver) pass `denoise=true` + a per-frame `uFrame` that **decorrelates** the RNG; the **offline/golden path keeps `denoise=false` + `uFrame=0`**, so the converged 1024-spp goldens stay **bit-identical** (verified: `diff_vs_golden=0.000000`). Heap 3->4 descriptors, PT root-sig table 3->4, constants 24->28.
- **Permanent guard.** New headless `--dxr-denoise`: render a high-spp **ground-truth reference**, then a 4-spp frame denoise-OFF and denoise-ON, and report the mean-abs channel error of each vs the reference. The **M9 gate fails** unless the denoiser moves the image *toward* ground truth (`err_on < err_off`) and cuts error to `< 0.7x`. Measured: **12.7 -> 4.60, ratio 0.36 (~64% noise cut)**, debug-clean -- so it genuinely denoises, not just blurs.
- **Consequences:** ray-traced mode is dramatically cleaner at the same 4 spp (a ~64% step toward converged) with negligible cost (one filter dispatch; M9 interactive PT still 170 FPS, debug-clean). Zero new dependency (fits "everything procedural"); determinism untouched (DXR is the non-gated presentation layer; goldens bit-identical). Fresh exe delivered. Future levers now unlocked: lower spp for more FPS, full SVGF temporal reprojection (needs motion vectors), or albedo demodulation for sharper textures -- deferred; this v1 already fixes the operator's complaint. Real DLSS remains an explicit operator-approved exception only (would need a new-dependency ADR + the no-assets waiver).
